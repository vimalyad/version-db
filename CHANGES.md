## Decisions & deviations

Record cross-cutting decisions here as they are made. Seeded with the ones already fixed in the plan:

- **Language/build:** Java 17, Maven, JUnit 5.
- **PAGE_SIZE:** fixed at **8192** bytes (`Constants.PAGE_SIZE`) in sub-phase 0.2. Matches PostgreSQL's default page size; never change for an existing database file.
- **Version storage:** MVCC uses **Option B — separate Version Store** (heap stays MVCC-agnostic), per `part3.md` §6.2 and `implementation.md` Phase 10.
- **Isolation default:** Snapshot Isolation (REPEATABLE READ semantics); READ COMMITTED via per-statement snapshots. SSI is out of scope.
- **Catalog statistics (Phase 4):** only the three system heaps from `part1.md` §6.2 (tables/columns/indexes) are persisted. Per-column `ColumnStats` are held in an in-memory cache (the execution engine recomputes them periodically per §6.4); persisted table counts live in `TableMeta`. Catalog metadata records are encoded with the shared `TupleCodec` against fixed per-heap schemas (column type stored as its enum name).
- **`TxStatus` introduced in Phase 7 (planned for 8.1):** the `CommitLog` public API returns a transaction-status type, so `txn/TxStatus` (IN_PROGRESS/COMMITTED/ABORTED, 2-bit codes) is created in Phase 7. The Transaction Manager (Phase 8) reuses this same enum rather than defining its own.
- **UPDATE uses independent RID chains (Phase 10):** the design doc (§6.8) suggests linking the new version's `prevVersionId` to the old version. We instead keep per-RID chains independent (`prevVersionId=-1` for new inserts in an UPDATE). Cross-chain linking would allow old readers scanning the new RID to traverse back and see old-version data — incorrect behavior. Keeping chains independent means old-version data remains visible only via the original RID, and new-version data is visible only via the new RID.
- **Vacuum heap reclamation is injected (Phase 11):** there is no global RID→`HeapFile` registry until the execution engine (Phase 15), so `Vacuum` takes a `HeapReclaimer` callback to physically free dead heap slots rather than referencing `HeapFile` directly. Tests and integration wire it to `HeapFile::deleteTuple`; the 3-arg `Vacuum` constructor defaults to a no-op reclaimer (version-store cleanup only). Also additive to Phase 10: `VersionRecord.prevVersionId` changed from `final` to `volatile` (so a survivor can be relinked as a new tail) and `MVCCManager.getVisibleVersion` treats a reclaimed (`null`) version as the end of the chain.
- **Optimizer simplifications (Phase 14):** an index scan keeps the *full* WHERE as its `postFilterPredicate` (the index range only narrows the candidate set; the executor re-checks the whole predicate) rather than computing the exact residual — simpler and always correct. Join-result cardinality uses coarse heuristics (equi-join selectivity ≈ 1/max(outerRows, innerRows), non-equi 0.1, cross 1.0) since per-column join statistics are not gathered. Plan nodes store estimates as `double`. The optimizer plans single-table SELECT/DELETE/UPDATE and multi-table SELECT joins; INSERT and DDL are handled directly by the execution engine (Phase 15), not optimized.
- **Version-store reseed on restart (Phase 16):** the MVCC `VersionStore` is in-memory and not persisted, so on reopen `getVisibleVersion` would find no versions and committed data would read as absent. `MiniDb` startup rebuilds it by **replaying the committed WAL history**: every committed INSERT not later deleted by a committed transaction becomes one committed "base" version, stamped with sentinel `xmin = FIRST_NORMAL_XID` — below every post-restart snapshot's `xmin`, so the visibility rule's `xid < snapshot.xmin` short-circuit makes those rows visible to all future transactions. (A heap scan is *not* sufficient: a DELETE/UPDATE only stamps the in-memory version store and the page LSN, so recovery's redo skips it and the dead bytes stay physically present — a heap scan would resurrect logically-deleted rows. CLRs belong to aborted losers, which never commit, so the commit-status filter excludes them.) Added the read-only accessor `Catalog.allTables()` (additive; no signature change) so startup can rebuild the non-persisted table→`HeapFile` and index-root→`BPlusTree` registries. **Known limitations:** the catalog and the heap page-chain `nextPageId` pointer are not WAL-logged, so DDL and multi-page heap growth are durable only via a clean `flushAll` at close (the crash-recovery test therefore keeps committed rows on already-allocated pages). The integrated `Vacuum` uses the no-op heap reclaimer (version-store GC only), since a global page→table map for physical slot reclamation does not exist.
