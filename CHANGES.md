## Decisions & deviations

Record cross-cutting decisions here as they are made. Seeded with the ones already fixed in the plan:

- **Language/build:** Java 17, Maven, JUnit 5.
- **PAGE_SIZE:** fixed at **8192** bytes (`Constants.PAGE_SIZE`) in sub-phase 0.2. Matches PostgreSQL's default page size; never change for an existing database file.
- **Version storage:** MVCC uses **Option B — separate Version Store** (heap stays MVCC-agnostic), per `part3.md` §6.2 and `implementation.md` Phase 10.
- **Isolation default:** Snapshot Isolation (REPEATABLE READ semantics); READ COMMITTED via per-statement snapshots. SSI is out of scope.
- **Catalog statistics (Phase 4):** only the three system heaps from `part1.md` §6.2 (tables/columns/indexes) are persisted. Per-column `ColumnStats` are held in an in-memory cache (the execution engine recomputes them periodically per §6.4); persisted table counts live in `TableMeta`. Catalog metadata records are encoded with the shared `TupleCodec` against fixed per-heap schemas (column type stored as its enum name).
- **`TxStatus` introduced in Phase 7 (planned for 8.1):** the `CommitLog` public API returns a transaction-status type, so `txn/TxStatus` (IN_PROGRESS/COMMITTED/ABORTED, 2-bit codes) is created in Phase 7. The Transaction Manager (Phase 8) reuses this same enum rather than defining its own.
- **UPDATE uses independent RID chains (Phase 10):** the design doc (§6.8) suggests linking the new version's `prevVersionId` to the old version. We instead keep per-RID chains independent (`prevVersionId=-1` for new inserts in an UPDATE). Cross-chain linking would allow old readers scanning the new RID to traverse back and see old-version data — incorrect behavior. Keeping chains independent means old-version data remains visible only via the original RID, and new-version data is visible only via the new RID.
- **Vacuum heap reclamation is injected (Phase 11):** there is no global RID→`HeapFile` registry until the execution engine (Phase 15), so `Vacuum` takes a `HeapReclaimer` callback to physically free dead heap slots rather than referencing `HeapFile` directly. Tests and integration wire it to `HeapFile::deleteTuple`; the 3-arg `Vacuum` constructor defaults to a no-op reclaimer (version-store cleanup only). Also additive to Phase 10: `VersionRecord.prevVersionId` changed from `final` to `volatile` (so a survivor can be relinked as a new tail) and `MVCCManager.getVisibleVersion` treats a reclaimed (`null`) version as the end of the chain.

### Phase 11 — Vacuum / GC  (branch: phase-11-vacuum)
- [2026-06-21] 11.1 — `txn/Vacuum`: the reclaim predicate `isReclaimable(version, horizon)` = `xmax != 0 AND xmax < horizon AND CommitLog.isCommitted(xmax)` (per `part3.md` §8.2) plus `currentHorizon()` delegating to `TransactionManager.getOldestActiveXid()`. A live head (`xmax==0`), a deleter at/after the horizon, and an uncommitted/aborted deleter are all kept. Added `VersionStore.chainHeadsSnapshot()` (read-locked copy of the `RID→headVersionId` map) so the upcoming sweep can iterate every chain without holding the store lock. Files: `txn/Vacuum.java`, `txn/VersionStore.java`, `test/.../txn/VacuumTest.java`.
- [2026-06-21] 11.2 — GC pass: `Vacuum.runOnce()` computes the horizon once and sweeps every chain; `gcChain` walks each chain from the tail (oldest) toward the head, removing the contiguous run of reclaimable versions and stopping at the first version that must be kept (`part3.md` §8.3), then detaches the surviving oldest version as the new tail (`prevVersionId = -1`). **Cross-phase additive changes** to Phase 10 code: `VersionRecord.prevVersionId` is now `volatile` (was `final`) so a survivor can be relinked; `VersionStore.remove(versionId)` tombstones a reclaimed record to `null` while keeping its slot so all other version ids stay stable; `MVCCManager.getVisibleVersion` now treats a `null` (reclaimed) version as the end of the chain. Removing only a contiguous, monotonically-reclaimable tail run keeps concurrent reads correct: a reader that already followed a link into the removed run hits a tombstone and stops, and every version there is invisible to it anyway. Files: `txn/Vacuum.java`, `txn/VersionStore.java`, `txn/VersionRecord.java`, `txn/MVCCManager.java`, `test/.../txn/VacuumTest.java`.
- [2026-06-21] 11.3 — Physical heap reclamation + background scheduler. New `txn/HeapReclaimer` functional interface (`freeSlot(rid)`, plus a `NO_OP`): when a chain is fully dead (the head version itself is reclaimable, so the row is permanently deleted and past the horizon) `gcChain` now calls `heapReclaimer.freeSlot(rid)` to physically tombstone the heap slot (`HeapFile.deleteTuple`) and `VersionStore.removeHead(rid)` to stop tracking the RID. `Vacuum` gains a 4-arg constructor taking the reclaimer (the 3-arg one defaults to `NO_OP`), and a background daemon scheduler `start(periodMillis)`/`stop()`/`isRunning()` that calls `runOnce()` on a fixed period (transient `runOnce` failures are swallowed so the worker survives to the next tick). Tests cover heap-slot freeing for a dead row, slot retention for a live row, automatic reclamation by the background thread, and a long-running transaction pinning the horizon to block reclamation until it commits. Files: `txn/HeapReclaimer.java`, `txn/Vacuum.java`, `txn/VersionStore.java`, `test/.../txn/VacuumTest.java`. **Phase 11 complete** (214 tests green).
